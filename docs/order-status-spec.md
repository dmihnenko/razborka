# Спецификация: расширенная модель статусов заказа

> Черновик-промпт для реализации. **Правь прямо в этом файле** (этапы, авто-правила, открытые вопросы),
> после согласования по нему делаю миграцию + код.

## Текущее состояние (как есть)
- Статусы: `new` · `in_progress` · `completed` · `cancelled` (CHECK-constraint на `parts_orders.status`).
- Канбан-доска заказов уже есть (3 колонки, drag-and-drop) — `PartsOrders.tsx`.
- Таймлайн заказа есть (из `parts_activity_log`) — `PartsOrderDetails.tsx`.
- ТТН создаётся вручную → регистрируется посылка (`createShipment`), Telegram-уведомление при отправке.
- Трекинг НП: коды `9/10/11` = доставлено/получено, `14/102/103/108` = проблема/возврат.
- Инвентарь: при `completed` позиции должны становиться `sold`; при `cancelled`/возврате → `available`.

## Предлагаемая модель (6 этапов + отмена)

| # | Статус (код) | Название | Что значит | Инвентарь |
|---|---|---|---|---|
| 1 | `new` | Новый | Заказ создан, позиции добавляются | — |
| 2 | `assembling` | Сборка | Подбираем/комплектуем заказ | позиции → **резерв** |
| 3 | `ready` | Готов | Собран, ждёт отправки/выдачи | резерв |
| 4 | `shipped` | Отправлен | Создана ТТН / передан в доставку | резерв |
| 5 | `delivered` | Доставлен | НП подтвердила получение (коды 9/10/11) | резерв |
| 6 | `completed` | Завершён | Деньги получены, сделка закрыта | позиции → **продано** |
| ✕ | `cancelled` | Отменён | Отмена с любого этапа | позиции → **в наличие** |

## Авто-правила переходов
- **new → assembling** — при добавлении первой позиции (или вручную). **Авторезерв** всех позиций заказа.
- **assembling → ready** — вручную («Заказ собран»).
- **ready/assembling → shipped** — **авто** при создании ТТН (уже есть хук `onTtnCreated`).
- **shipped → delivered** — **авто** из трекинга НП (коды 9/10/11) при обновлении статуса посылки (раздел «Доставка» / массовая проверка).
- **delivered → completed** — вручную («Деньги получены») *(см. открытые вопросы — возможно авто)*.
- **любой → cancelled** — вручную; возврат позиций в наличие.

## Эффекты на инвентарь (когда позиция меняет статус)
- **Резерв** — на `assembling` (и держится до завершения/отмены). Нужно поле учёта (статус позиции `reserved`).
- **Продано** — на `completed` (фиксируем `sold_price`, `sold_at`, `exchange_rate_at_sale`).
- **В наличие** — на `cancelled` и при возврате (sold/reserved → available).

## Открытые вопросы (ответь / поправь)
1. **Все 6 этапов нужны?** Или объединить — напр. убрать `ready` (сразу `assembling → shipped`)? Или убрать `delivered` (доставка видна в разделе «Доставка»)?
2. **Когда позиция «продано»** — на `delivered` (по факту получения) или на `completed` (по факту денег)? Для предоплаты и наложки логика разная.
3. **delivered → completed**: вручную или авто (напр. авто-завершение через N дней после доставки, или сразу при доставке если была предоплата)?
4. **Самовывоз** (без НП): нужен ли путь `ready → completed` напрямую, без `shipped/delivered`?
5. **Названия** статусов — устраивают? (Сборка/Готов/Отправлен/Доставлен/Завершён.)
6. **Авторезерв**: резервировать на `assembling` (как предложено) или сразу при добавлении позиции в `new`?

## План реализации (после согласования)
1. **Миграция БД**: новый CHECK-constraint со списком статусов; маппинг существующих данных (`in_progress → assembling`, остальные без изменений). Применяю сам через Management API.
2. **Воркфлоу-функция** `updatePartsOrderStatus`: эффекты на инвентарь по каждому статусу (резерв/продано/в наличие).
3. **UI статусов**: `getPartsOrderStatusText`, бейджи/чипы, кнопки переходов на странице заказа (последовательная цепочка + «Отменить»).
4. **Канбан**: колонки под новые статусы (свернуть на мобиле; группировать длинную цепочку).
5. **Авто-delivered**: в обновлении статуса посылки (`refreshShipmentStatus`) — если код 9/10/11, перевести связанный заказ в `delivered` (+ запись в таймлайн).
6. **Дашборд-воронка**: обновить под новые этапы.
7. **i18n** ru/uk, build/lint/паритет, проверка вживую.

## Риски
- Поведенческое изменение на живых заказах — миграция существующих статусов делается осторожно, без потери данных.
- Авто-`delivered`/`completed` — не должны срабатывать неожиданно; начинаем с минимума авто-правил, остальное вручную.
- Резерв требует корректного снятия при отмене/возврате/удалении заказа.
