## Порядок действий

1. **Бэкап.** Supabase → Database → Backups (или `pg_dump`). Без бэкапа не продолжать.
2. **Discovery.** Открой `remove_sto.sql`, выполни закомментированные запросы из
   **ШАГ 0** в SQL Editor. Сверь, какие СТО-таблицы реально есть в твоей базе и
   сколько СТО-строк в общих таблицах. Имена таблиц в проде могли отличаться.
3. **ШАГ 1 (обязательное).** Выполни блок `BEGIN … COMMIT` из ШАГ 1 — удаляет
   СТО-таблицы (записи, заказ-наряды, услуги, счета, сотрудники, компании СТО),
   чистит СТО-подписки/заявки/роли. Если таблицы `parts` (legacy-склад СТО) у тебя
   нет или она относится к разборке — оставь строку 1.3 закомментированной.
4. **ШАГ 2 (опциональное).** Гибридные таблицы `customers`/`vehicles` и столбцы
   `sto_company_id`. Раскомментируй и выполни **только после проверки**, что там
   нет нужных данных. Снятие столбцов `sto_company_id` безопасно — фронтенд их
   больше не использует.
5. **После.** Повтори ШАГ 0.1 (таблиц СТО быть не должно). Проверь RLS-политики и
   функции на остаточные ссылки на удалённые таблицы (ШАГ 0.2) — удали вручную.

## Что НЕ трогаем

- Общие таблицы: `roles` (минус СТО-роли), `user_profiles`, `user_roles`,
  `subscriptions`/`company_subscriptions` (минус `company_type='sto'`),
  `support_*`, `trash_bin`, `personal_vehicles`, `vehicle_share_links`, `app_settings`.
- Все `parts_*` (разборка).
- Edge Functions — общие, СТО-специфичных нет, деплой не нужен.

## Откат

Полного отката нет (DROP необратим). Восстановление — только из бэкапа из п.1.
Поэтому бэкап обязателен.
