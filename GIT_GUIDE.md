# Git Workflow для проекта TSP-V2

## Текущий статус
✅ Git репозиторий инициализирован
✅ Создан первый коммит с восстановленными заказ-нарядами

## Основные команды

### Просмотр статуса
```bash
git status
```

### Добавление изменений
```bash
# Добавить все изменения
git add .

# Добавить конкретный файл
git add src/pages/WorkOrders.tsx

# Добавить несколько файлов
git add src/pages/*.tsx
```

### Создание коммита
```bash
git commit -m "Описание изменений"
```

### Просмотр истории
```bash
# Краткая история
git log --oneline

# Подробная история
git log

# История с графиком
git log --oneline --graph --all
```

### Откат изменений

#### Откат изменений в рабочей директории (до git add)
```bash
# Откатить все файлы
git checkout .

# Откатить конкретный файл
git checkout src/pages/WorkOrders.tsx
```

#### Отмена git add (убрать из staging)
```bash
# Убрать все файлы из staging
git reset

# Убрать конкретный файл
git reset src/pages/WorkOrders.tsx
```

#### Откат последнего коммита (с сохранением изменений)
```bash
git reset --soft HEAD~1
```

#### Откат последнего коммита (без сохранения изменений)
```bash
git reset --hard HEAD~1
```

#### Откат к конкретному коммиту
```bash
# Посмотреть хеш коммита
git log --oneline

# Откатиться (с сохранением изменений)
git reset --soft <хеш_коммита>

# Откатиться (без сохранения)
git reset --hard <хеш_коммита>
```

### Создание веток
```bash
# Создать новую ветку
git branch feature/new-feature

# Переключиться на ветку
git checkout feature/new-feature

# Создать и сразу переключиться
git checkout -b feature/new-feature
```

### Слияние веток
```bash
# Переключиться на основную ветку
git checkout master

# Слить изменения из другой ветки
git merge feature/new-feature
```

## Рекомендуемый workflow

### 1. Перед началом работы
```bash
git status  # Проверить текущее состояние
```

### 2. После внесения изменений
```bash
git add .
git commit -m "feat: Добавлена функция автоназначения работника"
```

### 3. Периодически делайте коммиты
- После завершения функции
- После исправления бага
- Перед началом большого рефакторинга

## Типы коммитов (Conventional Commits)

- `feat:` - новая функция
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `style:` - форматирование кода
- `refactor:` - рефакторинг
- `test:` - добавление тестов
- `chore:` - рутинные задачи

## Примеры

```bash
# Новая функция
git commit -m "feat: Добавлена переназначение работника в заказ-нарядах"

# Исправление бага
git commit -m "fix: Исправлена ошибка импорта в Customers.tsx"

# Рефакторинг
git commit -m "refactor: Упрощен код PartsManager"
```

## Восстановление файла из предыдущего коммита

```bash
# Восстановить файл из последнего коммита
git checkout HEAD -- src/pages/WorkOrders.tsx

# Восстановить из конкретного коммита
git checkout <хеш_коммита> -- src/pages/WorkOrders.tsx
```

## Полезные команды

```bash
# Посмотреть изменения
git diff

# Посмотреть изменения конкретного файла
git diff src/pages/WorkOrders.tsx

# Посмотреть что в staging
git diff --staged

# Отменить все незакоммиченные изменения
git reset --hard HEAD
```

## Важно!

- Всегда делайте коммит перед большими изменениями
- Используйте понятные сообщения коммитов
- Не коммитьте `node_modules/`, `.env` и другие временные файлы (они в .gitignore)
- Периодически проверяйте `git status`
