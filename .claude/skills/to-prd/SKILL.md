# PRD из контекста (этот проект)

Преврати уже накопленный контекст (разговор + код) в PRD. **Не интервьюируй пользователя — синтезируй то, что уже знаешь** (для стресс-теста плана вопросами есть `grill-me`/`grill-with-docs`).

## Перед написанием
1. **Разведка кода.** Прочитай текущее состояние затронутой области, ближайшую похожую фичу (`customers`/`appointments`/`vehicles`), применяй доменный словарь проекта; учитывай зафиксированные решения (память, `CLAUDE.md`).
2. **Шов тестирования.** Определи, где фича будет проверяться — предпочитай существующие интеграционные точки, самый высокий уровень шва; подтверди с пользователем.

## Шаблон PRD (markdown в `docs/<feature>.md`)
- **Проблема** — от лица пользователя.
- **Решение** — от лица пользователя.
- **User stories** — нумерованный список «Как [роль], я хочу [возможность], чтобы [польза]».
- **Решения по реализации** — слои (`types → services → hooks → components/pages`), таблицы/RLS-миграции, интерфейсы, паттерны; код — только если критичен для решения.
- **Решения по тестам** — что и на каком шве покрываем.
- **Вне охвата** — явные границы.
- **Заметки** — доп. контекст, риски (мультитенантность, инвалидация кэша, безопасность RLS).

Держи PRD синхронным с конвенциями репо; крупную/неоднозначную фичу сначала прогоняй через `tech-lead`.
