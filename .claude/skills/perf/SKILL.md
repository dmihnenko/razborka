# Производительность (этот проект)

Сначала измерь, потом оптимизируй. Не делай микро-оптимизаций без подтверждённой проблемы.

## Бандл / загрузка
- Сборка: `npx vite build` — смотри размеры чанков. Есть предупреждение «chunks > 500 kB» (крупный — `index`, `Analytics`).
- Страницы уже грузятся через `lazy(() => import(...))` в `App.tsx` — **повторяй этот паттерн** для новых тяжёлых страниц.
- Тяжёлые библиотеки (графики в `Analytics`, и т.п.) — держи в lazy-чанках, не тяни в общий `index`. При необходимости настрой `build.rollupOptions.output.manualChunks` в `vite.config.ts`.
- PWA (`vite-plugin-pwa`) кэширует ассеты — учитывай при изменении стратегии загрузки.

## Запросы к данным (Supabase / react-query)
- **N+1 — главный риск.** В репо есть примеры (подсчёт авто на клиента в `customersService`). Где возможно — один запрос с агрегацией/вложенным select вместо цикла запросов.
- Выбирай только нужные колонки в `select(...)`, а не всегда `*`, для тяжёлых списков.
- react-query: адекватный `staleTime`/`gcTime`, чтобы не перезапрашивать без нужды (см. `useUserProfile` — кэш в localStorage + 4ч staleTime). Точные `queryKey`, чтобы кэш переиспользовался.
- Не плоди параллельные одинаковые запросы — выноси в общий хук.

## Рендер React
- Большие списки/доски — избегай лишних перерисовок: стабильные ключи, `useMemo` для производных данных (как `kanbanByStatus`, `weekDays` в `AppointmentsBoard`).
- Выноси тяжёлые вычисления из рендера в `useMemo`; колбэки, передаваемые в мемоизированные дети — `useCallback`.
- Не оптимизируй преждевременно — проверь React DevTools Profiler, что перерисовка реально дорогая.

## Проверка
- Сравни размер бандла до/после (`vite build`). Назови конкретные цифры.
- Убедись, что поведение не изменилось и сборка зелёная.
