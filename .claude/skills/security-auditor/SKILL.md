# Аудит безопасности (этот проект)

Цель — найти реальные дыры в авторизации и изоляции данных, а не теоретические. Этот проект — мульти-тенант CRM: данные разделены по `sto_company_id` / `parts_company_id`, роли — `admin`, `sto_owner`, `sto_worker`, `parts_owner`, `parts_worker`, `user`. Главный риск — утечка данных между компаниями.

## Что проверять в первую очередь

### 1. RLS — основная линия обороны
- **Каждая** таблица с данными компании ДОЛЖНА иметь RLS включённым и политику, фильтрующую по компании пользователя. Клиентский `.eq('sto_company_id', ...)` — НЕ защита, его можно обойти; защищает только RLS.
- Проверь, что политики используют принадлежность пользователя к компании (через `user_profiles`/`user_roles`), а не просто `auth.uid() is not null`.
- `INSERT`/`UPDATE`/`DELETE` политики так же важны, как `SELECT` — частая дыра: открытый SELECT с RLS, но INSERT без `WITH CHECK`.
- Публичные share-ссылки (`vehicle_share_links`, `/public/*`) — отдельный путь: проверь, что они отдают только то, что должно быть публичным, и валидируют код/срок.

### 2. Edge Functions (`supabase/functions/`)
- Привилегированные функции (`create-user`, `delete-user`, `reset-password`, `update-username`) используют `SERVICE_ROLE_KEY` — он обходит RLS. КРИТИЧНО: в начале функции проверять, что вызывающий имеет право (как в `create-user`: достаёт роли вызывающего и проверяет `canCreateUsers`). Без этой проверки любой авторизованный юзер = админ.
- `SERVICE_ROLE_KEY` только из `Deno.env.get(...)`, никогда не в клиенте/репозитории.
- CORS: `ALLOWED_ORIGINS` должен быть allowlist (как сейчас), не `*`, особенно для функций с service role.
- Валидируй вход (`req.json()`) — не доверяй `role_ids`, `company_id` от клиента без проверки прав.

### 3. Секреты
- `VITE_*` переменные попадают в бандл и видны всем — там ТОЛЬКО публичный anon key и URL. `SERVICE_ROLE_KEY`, ImgBB private ключи и т.п. в `VITE_*` быть НЕ должны.
- Grep по репозиторию на захардкоженные ключи/токены/пароли перед релизом.

### 4. Авторизация на клиенте
- Клиентские проверки роли (`useIsAdmin`, скрытие кнопок) — это UX, не безопасность. За каждой такой проверкой должна стоять RLS/Edge-проверка на сервере.
- Роуты: убедись, что приватные страницы реально недоступны без нужной роли, а не просто скрыты в меню.

### 5. PWA / заголовки
- Service Worker не должен кешировать ответы с приватными данными в долгоживущие кеши (проверь `runtimeCaching` в `vite.config.ts`: auth/storage — NetworkFirst, коротко; не CacheFirst).
- Mixed content: всё по HTTPS (есть `upgrade-insecure-requests` в `index.html`).

## Метод
1. Перечисли таблицы (миграции / схема) → для каждой: есть ли RLS + корректная политика по компании?
2. Перечисли Edge Functions → у каждой с service role есть проверка прав вызывающего?
3. Grep секретов и `VITE_` на чувствительное.
4. Сопоставь клиентские гейты ролей с серверными.
5. Выдай находки по severity (critical/high/medium), каждую — с файлом/строкой и конкретным фиксом. Не выдумывай дыры, которых нет.

## Чек перед сдачей
- Каждая находка воспроизводима и привязана к коду.
- Для critical/high предложен конкретный патч (SQL-политика или проверка в функции).
- Не сломаны существующие легитимные сценарии доступа.
