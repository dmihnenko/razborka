# QA / тестирование (этот проект)

## Инструменты и расположение
- **Unit/компонентные:** Vitest, `environment: 'jsdom'`, `globals: true`, setup — `src/test/setup.ts`, моки — `src/test/mocks/`. Файлы тестов лежат рядом с кодом: `*.test.ts(x)` (напр. `src/services/partsService.test.ts`, `src/hooks/useAuth.test.ts`, `src/components/ErrorBoundary.test.tsx`).
- **E2E:** Playwright, конфиг `playwright.config.ts`, сценарии в `e2e/` (есть `e2e/smoke.spec.ts`).
- Команды: `npm test` (vitest run), `npm run test:watch`, `npm run test:coverage`, `npm run e2e`, `npm run e2e:ui`.

## Как писать
1. **Разведка.** Открой соседний тест того же типа и повтори стиль (импорты, моки Supabase, структуру `describe/it`).
2. **Изоляция данных.** Supabase мокается — не ходи в реальную БД из unit-тестов. Используй существующие моки из `src/test/mocks/`; если нужного нет — добавь по образцу.
3. **Что покрывать в первую очередь:** чистую логику в `services/` (расчёты сумм, фильтры статусов, мультитенантные проверки — что без `sto_company_id` возвращается пусто), затем хуки и критичные компоненты.
4. **Гранулярность:** один `it` — одно поведение; имена на русском допустимы, как в репо. Проверяй и happy-path, и граничные случаи (пусто, ошибка, null).
5. **E2E** — только ключевые пользовательские потоки (логин, создание заявки, смена статуса). Держи их стабильными (явные ожидания, не sleeps).

## Правила
- Не меняй продакшн-код ради «удобства теста» без необходимости — тестируй как есть.
- Прогон: `npm test` зелёный. Если правил поведение — обнови затронутые тесты и объясни почему.
- Не вводи новые unused-locals (в репо есть долг по `tsc`, но новый не добавляй).
- Для ручной проверки фичи в реальном приложении используй скил `verify` / `run` (запуск `npm run dev`), а не только автотесты.
