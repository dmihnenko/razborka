# Диагностика (этот проект)

Шесть фаз. Не чини вслепую — сначала собери сигнал.

## 1. Петля обратной связи (ключевой навык)
**Если есть быстрый, детерминированный, запускаемый агентом pass/fail-сигнал на баг — причина найдётся.** Построй его прежде всего:
- падающий Vitest (`npm test`) или Playwright (`npm run e2e`) — предпочтительно;
- минимальный repro-скрипт/функция; для данных — SQL-запрос к боевой/локальной БД через Management API;
- для UI — узкий повтор в `npm run dev`;
- дифф-петля «работает на коммите A, падает на B» (`git bisect` мысленно).
Доведи петлю до быстрой, чистой по сигналу и детерминированной.

## 2. Воспроизведи
Убедись, что петля ловит ИМЕННО заявленный пользователем сбой, а не соседний симптом.

## 3. Гипотезы
Сформулируй 3–5 ранжированных **фальсифицируемых** гипотез ДО правок. Каждая предсказывает конкретное наблюдаемое. Покажи их пользователю — он может переранжировать по доменному знанию (RLS, react-query кэш, мультитенантность, цикл вендор-чанков).

## 4. Инструментируй
Привяжи пробы к предсказаниям. Логи/брейкпоинты помечай тегом для лёгкой уборки. Для перформанса — сними базлайн (бандл, число запросов, N+1) ДО фикса.

## 5. Фикс + регресс-тест
Пиши тест на правильном шве — там, где баг возникает естественно (сервис/хук/RPC). Нет удобного шва → это сигнал архитектурного долга. Инвалидация react-query — частый корень (см. память о свежести данных).

## 6. Уборка + пост-мортем
Проверь, что исходный сценарий больше не падает; убери ВСЮ инструментацию; запиши, какое архитектурное изменение предотвратило бы баг (при ценности — в память).
