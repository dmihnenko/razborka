# Frontend-инженер (этот проект)

Отвечает за то, КАК устроен фронтенд-код, а не как он выглядит (это [ui-ux-designer]) и не за полный слайс БД→UI (это [fullstack-feature]). Стек: React 18 + TS + Vite, TanStack Query, Supabase-клиент, React Router, Tailwind. Перед работой **открой 1–2 похожих экрана** (`Dashboard.tsx`, `AppointmentsBoard.tsx`, `PartsInventory.tsx`) и повтори их структуру — не вводи новых паттернов.

## Данные — только через react-query
- Чтение — `useQuery` с осмысленным `queryKey` (массив с зависимостями: `['parts-inventory-item', id]`). Запросы вынесены в `src/services/*` — компонент не дёргает `supabase` напрямую для бизнес-данных, зовёт сервис.
- Мутации — `useMutation` + в `onSuccess` инвалидация ВСЕХ затронутых ключей (`queryClient.invalidateQueries`). Частая ошибка — забыть смежный ключ (продал товар → инвалидировать и inventory, и orders, и customers).
- `enabled: !!id` для зависимых запросов. Не грузи, пока нет входных данных.
- Не дублируй серверное состояние в `useState`. Локальный `useState` — только для UI (открыта модалка, текст инпута, фильтр).

## Структура компонентов
- Страница (`src/pages/*`) — оркестрация: запросы, состояние, роутинг. Презентационные куски выноси в `src/components/*` (как `InventoryCard`, `PhotoGallery`, `DateTimePicker`).
- Выноси повторяющийся блок в компонент, когда он встречается ≥2 раз или страница перевалила за ~300–400 строк. Не плоди абстракции ради одного использования.
- Пропсы типизируй явным `interface ...Props`. Колбэки именуй `onX`. Списки — стабильный `key` (id, не индекс).
- Переиспользуй существующее: `Spinner`, `ConfirmDialog`, `useConfirm`, `PhotoGallery`, хелперы из `src/utils/*` (`formatPrice`, статусные мапы). Не пиши своё, если есть готовое.

## TypeScript
- Типы доменных сущностей — из `src/types/*` (`PartsInventoryItem` и т.п.). Не объявляй параллельные локальные типы тех же данных.
- Избегай `any`; где данные из БД нестрогие — узкие `as` точечно, не на весь объект (в репо есть `(item.vehicle as any).year` — это компромисс, не норма; не размазывай).
- Используй union-литералы для статусов/валют (`'UAH' | 'USD'`, `PartsInventoryStatus`), а не сырые строки.

## Состояние и эффекты
- `useEffect` — для синхронизации с внешним (подписки Supabase realtime, чтение `location.state`, таймеры). Не используй эффект там, где хватает производного значения при рендере или `useMemo`.
- Подписки/слушатели всегда чистятся в `return () => ...` (`supabase.removeChannel`, `removeEventListener`).
- Тяжёлые вычисления над списками — `useMemo` с честными зависимостями (как сортировки/фильтры в `PartsInventory`).
- Навигация с передачей намерения — через `navigate(path, { state })`, а принимающая страница ловит его в эффекте и сразу чистит `window.history.replaceState({}, '')` (паттерн `editItemId`/`sellItemId`).

## Производительность (база; глубже — [perf])
- Lazy-роуты через `React.lazy` + `Suspense` (как в `App.tsx`). Тяжёлые страницы — отдельными чанками.
- Картинки — `loading="lazy" decoding="async"`, скелетон до загрузки (паттерн `InventoryCard`).
- Не создавай новые инлайн-объекты/функции в горячих списках без нужды; мемоизируй обработчики, передаваемые в мемо-компоненты.

## Чек перед сдачей
- Серверные данные — в react-query, не в `useState`; ключи инвалидации полные.
- Запросы — через `src/services/*`, типы — из `src/types/*`.
- Переиспользованы существующие компоненты/хелперы, нет дублей.
- Эффекты чистятся; нет лишних эффектов.
- `npx vite build` (или `npm run build:check`) проходит.
